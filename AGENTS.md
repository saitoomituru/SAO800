## このプロジェクトの性質

- **コアビジョン**: OND800（演者のコックピット）が戦場で最大のパフォーマンスを発揮できるよう、母艦（Mac / OBS）側が重演算・兵装ビルド・管制を全部引き受ける。
- **OBSプラグイン**として実装することで、OBSのMetalレンダラー・Metalコンポジットに寄生し、重い処理をハードウェアにオフロードする。
- **演算するんじゃなくて指揮する**。SAO800がやることは「どの座標にどの素材を置くか」を決めてOBSに渡すだけ。コンポジット処理はOBSに丸投げ。
- FMP/FMS Advの関係と同じ。OND800が届かない演算リソースをSAO800が巻き取る。
- 思想駆動・実験駆動。要件定義から厳密に積まず、ビジョンベースでスモールに実装してアジャイルする。
- **開発リソースは投げ銭とジャンク箱**。停滞・放置期間が発生しても「プロジェクト停止」と判断しない。再開時に `/notes` を読んで状況再把握すること。

### SAO800の役割 ── スクラブナース / 手元見習いモデル

SAO800の立場は「アークエンジェル（後方支援母艦）」と呼ぶが、
より現場感のある比喩として **スクラブナース** / **手元見習い** モデルを採用する。

```
スクラブナース（SAO800）
  └─ 術者が「次」と言う前に器具を手に持って待ってる
  └─ 術野を汚さない（SaaSへのNSFWコンテキストを事前処理）
  └─ 使い終わった器具を受け取って次の準備（フェールオーバー管制）

術者（OND800）
  └─ 判断と執行だけに集中
  └─ スロット番号を投げるだけ
```

手元見習いモデルで言えば、**親方（OND800）が「3番」と言った瞬間に工具が手の中にある状態**を作るのがSAO800の仕事。

このモデルが意味すること:

- **先読みして準備する** — OND800が要求する前に、次に必要になるアセット・ストリーム・フォールバックを用意しておく。リアクティブではなくプロアクティブ。
- **術野を汚さない** — NSFWコンテキストはSAO800がローカルで処理してから渡す。OND800の判断フローにノイズを混入させない。
- **術者の集中を守る** — OND800（演者）は「スロット番号を投げる」以上のことをSAO800との通信に使わない。残りの認知リソースはすべて演技・配信に向く。
- **受け取って次を準備する** — フェールオーバー・プラットフォーム切替・ログ収集はSAO800が黙って処理する。術者に「今SaaS死んでるよ」と報告するのではなく、切り替えてから事後通知する。

> **設計判断の基準**: 「これはSAO800がOND800の代わりに先回りして用意できるか？」。
> YES なら SAO800 に実装する。NO（演者の判断が必要）なら OND800 に実装する。

## アーキテクチャ原則（違反禁止）

1. **OBSに寄生する** — OBSのMetalレンダラーを自前で実装しない。OBS APIを使う。
2. **座標を渡すだけ** — コンポジット処理はOBSに丸投げ。SAO800がピクセルを触ることは原則ない。
3. **ローカルファースト** — SaaS依存フィーチャーはローカルフォールバックを必ず持つ。
4. **OND800はスロット番号だけ投げる** — SAO800側でスロット→アセット→OBS APIの変換を全部やる。
5. **モザイクは実装しない** — スタンプによるコンテキスト上書きがコンセプト。モザイクは「隠してる」メタ情報がNSFWを強化するため禁止。
6. **NSFWコンテキストはSaaS到達前にローカルで処理する** — SphereOS / LLaMaがギャグ変換してからClaudeに渡す。クリーンなコンテキストしかSaaSに出さない。
7. **コロニー落とし禁止** — 炎上増幅・攻撃的自動投稿を行う機能は実装しない。
8. **フェールオーバーは第一級機能** — SaaS障害対応をエッジケースとして後回しにしない。
